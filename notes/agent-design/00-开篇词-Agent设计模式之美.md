# 开篇词｜把一千个真实工程问题凝结成 28 个设计模式

**作者**：黄佳（新加坡科研局资深研发工程师，《Designing AI Agents》作者，ADPS 创始成员）

---

## 三代设计模式进化

| 代际 | 底层假设 | 控制流由谁决定 |
|---|---|---|
| GoF 23 模式 | 单进程 / 共享内存 / 确定性调用 | 写代码时固定死，运行时按写好的走 |
| 分布式系统模式 | 网络不可靠 / 节点会挂 | 控制流还是人写的，只是执行环境不可靠 |
| Agent 设计模式 | 模型生成内容非确定性 | 一部分控制流在运行时由模型实时生成 |

> GoF 的"确定性"本质不是"系统可靠"，而是**控制流在编译期就被固定**。
> 分布式的"不可靠"是执行环境不可靠，但控制流还是人写的。
> Agent 的"不稳定"是**连生成的内容本身都可能是非确定性的**，同一 prompt 两次可能出两个不同答案。

一句话概括：**GoF 解决"代码怎么组织"；分布式解决"服务怎么存活"；Agent 解决"模型怎么被约束地行动"。**

## 核心判断

> **Agent 的价值跟它累积的经验等比，跟它"今天有多聪明"反而关系不大。**

一个聪明的 Agent 容易做，但三个月后它有没有记住踩过的坑？有没有学会这个仓库的规范？有没有把成功流程沉淀成 Skill？有没有因为处理过 100 个 PR 就比第 1 个 PR 时更懂这个仓库？如果没有，它只是一个每天重新入职的聪明新人。

**与此相关的工程目标转变：**

> 以前写代码的工程师追求一次写对；现在做 Agent 工程的工程师追求出问题了能很快定位。

这和"价值在累积"是同一枚硬币的两面：积累经验才能快速定位，定位问题才能积累经验。

---

## 范式转变

传统软件：工程师在设计时决定控制流（输入一样，输出基本一样）

Agent 系统：控制流的一部分被交给模型在**运行时**决定——先看什么、查什么、调用哪个工具、要不要重试、要不要派子 Agent，这些都根据上下文动态生成。

**过去工程师写逻辑。现在工程师更重要的是设计决策边界。**

> The model spends; the harness budgets. （模型负责花，Harness 负责控制预算）

## 双轴正交框架

### Y 轴 · 认知功能（Agent 要做什么）
1. **感知** — 看见世界
2. **记忆** — 跨时间保存状态
3. **推理** — 做判断
4. **行动** — 改变外部世界
5. **反思** — 检查改进自己
6. **协作** — 多角色分工
7. **治理** — 被信任地放进企业流程

### X 轴 · 执行拓扑（能力如何被编排）
1. **链式** — 一步步走
2. **并行** — 多路同时看
3. **路由** — 从多条路里选
4. **循环** — 试了再改
5. **编排** — 一个角色交给另一个
6. **层级** — 主 Agent 拆给子 Agent

交叉形成 7×6 的工程坐标系，28 个模式在其中可以被定位、比较、组合、复盘。

### 执行拓扑区分（易混辨析）

| 拓扑对 | 核心区别 |
|---|---|
| 链式 vs 层级 | 链式是流水线；层级是主 Agent 拆给子 Agent，子 Agent 各管各的上下文边界，主 Agent 不一定掌握完整思考过程 |
| 编排 vs 层级 | 编排的主 Agent 全程持有完整上下文，分配任务后还要汇总；层级的子 Agent 相对独立运行 |
| 路由 vs 循环 | 路由是先判断再选路径；循环是先做，看结果，不满意就重试 |

### 三代模式：叠加而非替代

> Agent 越复杂，对传统工程能力要求越高。三代模式更像是叠加关系，而非替代关系。
>
> GoF 和分布式没有"失效"——它们仍然是地基，只是现在多了一层"最上层"的问题需要处理：Agent 运行时决策的非确定性。

### 工程目标的转变

| | 传统软件 | Agent 系统 |
|---|---|---|
| 目标 | 结构定义完 = 行为确定 | 结构只能提高行为稳定的**概率** |
| 控制流 | 编译期固定 | 一部分在运行时由模型生成 |
| 工程目标 | 绝对正确 | 在不确定性下保持系统可靠 |

## 28 个模式的本质：资源分配学

| 模式类别 | 分配的稀缺资源 |
|---|---|
| 感知模式 | 注意力预算 |
| 记忆模式 | 时间跨度 |
| 推理模式 | 成本、准确率和延迟 |
| 行动模式 | 副作用风险 |
| 反思模式 | 经验沉淀 |
| 协作模式 | 角色和上下文 |
| 治理模式 | 信任 |

> Agent architecture is bounded resource allocation under uncertainty.
> Agent 架构是在不确定性下分配有限资源。

## 课程结构（9 模块 / 44 篇）

1. **范式觉醒** — 为什么 GoF 不够用，双轴框架是什么，逆向五步法拆 Harness
2. **感知** — 上下文分诊、语义压缩、渐进发现、多模态融合
3. **记忆** — 短期上下文、长期记忆、失败经验、进度追踪
4. **推理** — 思维链、复杂度路由、并行探索、迭代假设验证
5. **行动** — 工具调度、规划执行、提示链、护栏三明治
6. **反思** — Generator-Critic、Skill Package、Experience Replay、Self-Heal Loop
7. **协作** — 层级委派、扇出聚合、对抗评审、交接链
8. **治理** — 审批门、爆炸半径、渐进承诺、可观测性
9. **组合** — Pattern Selection Card、六步选型法、Argus Final 完整系统

## 三种能力目标

1. **看穿系统** — Agent 不稳定时，不只是"模型不够强"，知道从哪个维度定位
2. **组合模式** — 真实业务是混合问题，单个模式只是零件，适配组合才叫架构
3. **设计成长机制** — 失败日记、技能包、经验回放、进度追踪，让 Agent 从第 1 天聪明长成第 365 天老练

## 补充：Harness vs Prompt

> Prompt 告诉模型"该怎么做"；Harness 保证"即使模型没完全按预期做，系统也能把结果拉回正轨"。

举例：要求 Agent 生成带注释的 Java 代码
- 只靠 prompt = 期待模型一次性听话
- 用 Harness = 把是否有注释变成**可检测、可修复、可回撤、可复验**的工程环节
