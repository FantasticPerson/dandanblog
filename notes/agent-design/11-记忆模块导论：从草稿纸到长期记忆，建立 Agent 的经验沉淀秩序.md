# 11｜记忆模块导论：从草稿纸到长期记忆，建立 Agent 的经验沉淀秩序

**作者**：黄佳

---

## 一句话脉络

感知模块结束，进入记忆模块。记忆不是"让 Agent 记住东西"这么简单——它真正要解决的是**防止长程任务在时间中漂移**。

---

## 《记忆碎片》的启示

电影主角患短期记忆障碍，新的记忆无法稳定变成长记忆。他靠外部线索（照片、纸条、纹身）把关键事实留在身体和物件上——给自己搭了一套外部记忆系统。

Agent 也一样。窗口一关、会话一结束，Agent 就变回对你和你的项目都不熟悉的陌生人。

**记忆设计 = 给 Agent 留下照片、纸条和纹身**

## 记忆要解决的真正问题

### 开篇案例：auth.py 重构 Agent

一个负责代码重构的 Agent，在前面几轮发现了 `UserSession` 和 `PermissionCache` 之间的循环依赖，横跨四个文件，建议先抽共享 `types.py`。

Session 3 结束时，进度摘要这样写：

```
已完成：
- 确认 UserSession 与 PermissionCache 存在循环依赖
- 建议先引入共享 types.py
待继续：
- 开始移动 UserSession 相关代码
```

Session 4 的 Agent 看到这句话，做出了一个"合理但危险"的解释：先把 UserSession 移到新模块，**再在移动过程中顺手补 types.py**。结果：43 个测试失败，CI 无法继续，项目回滚。

**丢失的最关键信息**：
- 循环依赖的**具体路径**
- 已经被排除的**错误做法**
- **禁止动作**：不要先移动 UserSession

### 长程 Agent 的五种迷失

| 迷失类型 | 表现 |
|---|---|
| **目标漂移** | 从"完成 auth.py 重构"变成"把这个 import error 修干净" |
| **状态漂移** | Agent 以为环境是 A，真实环境已经是 B |
| **错误放大** | 第一步误判 schema 字段，后面查询验证文档都顺着误判长出来 |
| **细节过载** | 围着一个 Warning Message 把预算耗完 |
| **完成幻觉** | todo 都打勾了当成目标达成 |

## 记忆是 PRA 循环的时间维度

PRA 循环（感知 → 推理 → 行动）解决的是"这一轮"的问题。

记忆解决的是**时间问题**：上一轮的感知、推理、行动，怎样影响下一轮？

---

## 四种记忆类型

| 类型 | 说明 | 对应模式 |
|---|---|---|
| **Working Memory** | 当前 context window 里正在用的信息 | 分层保留 |
| **Scratchpad（草稿纸）** | 当前任务里的工作台，暂存中间判断、工具结果分析、下一步计划 | 分层保留 |
| **Episodic / Semantic / Procedural** | 跨会话的长期记忆 | 进度追踪 / RAG / 技能包 |
| **Memory Trace** | 记忆的仪表盘：写入、取回、使用、淘汰的记录 | 各模式 |

### Scratchpad 是最容易被低估的

> 它是记忆工作台，不是档案馆。

工具调用刚回来，规则还没核完，几个方案还没排除，下一步还没定——这些先放工作台。任务收束后，再决定哪些扔掉、哪些写进度追踪、哪些进失败日记、哪些进长期知识库。

---

## 记忆的生命周期

```
context window（上下文窗口）
    ↓
scratchpad（草稿纸）
    ↓
structured trace（结构化追踪）
    ↓
long-term memory（长程记忆）
    ↓
retrieval / replay / forgetting（检索/重放/遗忘）
```

---

## 记忆模式组的四种模式

记忆组四讲 = **架、取、录、省**

| 模式 | 解决 | 关键字 |
|---|---|---|
| **分层保留**（第一讲） | 架 | 记忆的货架怎么搭 |
| **检索增强 RAG**（第二讲） | 取 | 语义记忆怎么取回来 |
| **进度追踪**（第三讲） | 录 | 情节记忆怎么沉淀 |
| **失败日记**（第四讲） | 省 | 失败经验怎么召回 |

---

## 记忆的四个工程问题

1. 什么信息只该留在 context window 里？
2. 什么信息该先放 scratchpad，等任务收束再决定去留？
3. 什么信息应该写进长期记忆，并在需要时准确召回？
4. 什么信息已经过期，应该被淘汰或降权？

## 记忆框架选型

| 框架 | 特点 |
|---|---|
| **Letta** | Agent 在 core / archival / recall 等记忆层之间自己搬运 |
| **Mem0** | 单遍层次抽取 + 多信号检索 |
| **Zep** | 时序知识图谱，擅长表示"事实随时间怎么变" |
| **Anthropic Memory Tool** | 底层，Agent 直接读写持久化记忆目录 |

> 建议不要同时上两套记忆系统。记忆有单一事实来源问题。

## 记忆不是存储的堆叠，而是经验的治理

### 记忆注入（Memory Injection）风险

当线索被选择性记录、被错误解释、甚至被故意篡改时，记忆会变成制造幻觉的机器。

未来 Agent 攻击可能包括：
- **记忆注入**：先错误线索写入外部记忆
- **记忆投毒**：污染 Agent 对用户、项目和历史任务的理解
- **记忆篡改**：让 Agent 沿着错误方向继续行动

**记忆需要在设计时就考虑**：写入什么、谁有权改、什么时候过期、下次如何召回、召回后是否真的被用上。

## Memory Trace：记忆的行车记录仪

出了问题，顺着这串追问追：

```
这次行动需要依赖哪段过去经验？
这段经验当时有没有被写入记忆？
如果写了，这次为什么没被召回？（没命中？被淘汰？召回后和场景对不上？）
如果没写，是哪个环节本该写却没写？
下一版，你会怎么改记忆 pipeline？
```

每环都可能出问题：
- 没有写入 → 沉淀问题
- 没有取回 → 检索问题
- 取回不用 → 推理或规划问题
- 用了还错 → 记忆质量或适用边界问题

---

## 感知 vs 记忆：边界在哪里？

- **感知**关心的是**空间**——这一轮会话里，Agent 看到了什么
- **记忆**关心的是**时间**——当前窗口关掉之后，怎么把有用的东西沉淀下来

一份长期保留的 CLAUDE.md，你说它是感知（因为决定这一轮看到什么）也对，说是记忆（因为跨会话留下来了）也对。**把边界想清楚，对设计系统很有帮助。**
