# Agent 应用开发实战手册

> 基于《Agent 设计模式之美》课程整理
> 核心思维：**不确定性是默认条件，不是边缘情况。模型负责花钱，Harness 负责管账。**

---

## 一、先问五个问题

动手前，拿这五个问题过一遍你的系统：

1. **七脉状态是什么？** — 感知/记忆/推理/行动/反思/协作/治理，每项评 None/Light/Heavy
2. **每个 Heavy 功能落在哪个拓扑？** — Chain / Route / Parallel / Orchestrate / Loop / Hierarchy
3. **主要错误传播路径是什么？** — 这直接带出测试策略
4. **哪些格子刻意留空？** — 没做是因为不需要还是忘了？
5. **Compound Error：** 单步 95% → 10 步 ≈ 60%，你的 Agent 走多少步？

---

## 二、Harness 第一：先立边界，再写功能

### 五个地基（选型或自建时必问）

| 地基 | 关键问题 |
|---|---|
| 主循环 | 用户输入→模型调用→工具结果→状态更新，有吗？ |
| 上下文管理 | 有限 context 里放什么、丢什么、压缩什么？ |
| 工具注册与执行 | 模型能调什么、不能调什么？ |
| 状态账本 | 长任务可恢复吗？出错能回放吗？ |
| 治理边界 | 权限、沙箱、审批、审计——有吗？ |

> **这 5 个是显式机制，还是靠 prompt 硬撑？**

### 三类工程性格（你的系统属于哪一类）

| 类型 | 最重脉 | 代表框架 |
|---|---|---|
| 开发者工具型 | 感知、行动、治理 | Claude Code, Aider |
| 个人助手型 | 记忆、行动、治理 | Hermes |
| 重执行/多 Agent 型 | 协作、行动、治理、可观测 | OpenHands |

## 三、感知模块：Agent 能看见什么

### 处理顺序

```
多模态融合（最前）→ 上下文分诊 → 语义压缩 → 渐进发现（最后）
```

**形态一开始就错了，后面全是徒劳。**

### 上下文分诊：P0/P1/P2/P3

| 级别 | 处理方式 | 预算 |
|---|---|---|
| **P0** | 直接进 context，必须看到 | 当前步骤不可缺的证据 |
| **P1** | 原文片段进 context | 高度相关的背景 |
| **P2** | 压缩成摘要后进入 | 有价值但不需原文的全部 |
| **P3** | 只挂 handle，不预加载 | 可能有用但当前不值 |

**核心：控制信息和模型之间的距离，不是删材料。**

### 语义压缩：三层逐级压

| 层级 | 触发条件 | 处理方式 |
|---|---|---|
| L1 | context 占用 60-70% | 清理冗长 tool output → 替换成占位符 |
| L2 | 不够了 | 合并进 Anchor |
| L3 | 窗口撑不住 | 考虑交接或开新 session |

**Anchor 必须保住五类信息：**
- Intent（用户要什么）
- Changes made（已经改了/调了什么）
- Decisions taken（已经做了哪些判断）
- **excluded_approaches（已经排除的方案——最容易被忽略）**
- Next steps（下一步做什么）

### 渐进发现：三阶段循环

```
Forage（广扫）→ Focus（聚焦）→ Deepen（深挖）
```

- Forage：grep/glob，只看路径和匹配行，几千 token
- Focus：读 5-10 个候选文件，建立局部理解
- Deepen：沿 1-2 条可疑链追，不能再铺开

**工程纪律：** 给 Agent atomic tools（grep/glob/read 分开），max_cycles = 3，记录 trace。

### Agentic Search vs RAG

| | Agentic Search | RAG |
|---|---|---|
| 适用 | 代码库 debug、调用链追踪 | 文档库、知识库、政策规则 |
| 找什么 | 结构关系（变量名、调用链） | 语义相似片段 |
| 优势 | 无影子仓库、实时精准 | 适合非代码文本 |

**选型判断：** 中小代码库 → Agentic Search；超大代码库 → 持久化索引；政策/规则 → RAG + Evidence Contract。

## 四、记忆模块：Agent 能记住什么

### 分层保留（五层货架）

| 层级 | 放什么 | 覆盖规则 | Token 预算 |
|---|---|---|---|
| 策略层 | 合规红线、安全边界 | 不可覆盖、Agent 不可写 | 1200 |
| 项目层 | CLAUDE.md、项目规范 | 可被策略覆盖 | 3000 |
| 用户层 | 偏好、配置 | 可被项目硬规则覆盖 | 1500 |
| 任务层 | 当前目标、里程碑 | 不可覆盖长期事实 | 5000 |
| 草稿纸层 | 当前工具结果、中间判断 | 不可直接污染长期记忆 | 2500 |

**写入路由：** 推理 → scratchpad → 任务收束 → 有证据则升层到用户/项目层。

### RAG：从"相关文档"到"可用证据"

**证据契约四件套：** source + version + scope + citation

**RAG 落地顺序：**
1. 先收集真实问题和正确证据，填证据表
2. 给知识源补 owner、版本、生效日期、权限
3. 定义 chunk schema、citation schema
4. 建 candidate index，用 golden questions 做回归
5. 通过后切 alias
6. 接入引用 + RetrievalTrace

**机械状态不走 RAG：** id、金额、账号等精确参数走工具 + SessionState。

### 进度追踪（三平面）

| 平面 | 管什么 | 谁维护 |
|---|---|---|
| 调度态（Workspace） | 任务 DAG，谁 ready/blocked | 编排器 |
| 叙事态（Narrative） | 锚、账、集——目标、理由、上下文 | LLM + 系统 |
| 机械态（State） | 精确参数，带 Provenance | **程序维护** |

**三个调度收敛器：**
- **复诵**：隔固定步数把全局计划推回上下文
- **漂移哨兵**：定期问"你现在做的事还跟原目标有关吗？"
- **验证闸门**：每个里程碑验收，不通过不许进下一阶段

### 失败日记（6 层结构）

| 层 | 解决的问题 |
|---|---|
| 失败边界 | 算不算值得沉淀的失败（hard/gate/semantic/safety） |
| 失败分类 | 属于哪种可重复的失败模式 |
| 证据包 | 当时到底发生了什么（三平面快照） |
| 根因与补救 | 为什么会错、系统怎么改、下次怎么防 |
| 召回触发器 | 下次什么时候应该想起它 |
| 留存与审查 | 谁确认、保留多久 |

**关键区分：** 错误日志服务一次维修，失败日记服务下一次飞行。

## 五、错误处理与质量

### Compound Error 公理

> 单步 95% → 10 步 ≈ 60%，20 步 ≈ 36%

四条应对策略：

| 策略 | 做法 |
|---|---|
| 减少步数 | 能一次可靠完成的不拆成十步 |
| 提高单步质量 | 上下文给准、工具描述写清、schema 约束明确 |
| 加 verification | 中间状态就校验，不等最终结果 |
| fail fast | 明显错了就停，不带着脏状态继续 |

### 三种不确定性

| 类型 | 含义 | 应对 |
|---|---|---|
| 输出不确定 | 同 prompt 跑十次得十种答案 | 熔断器、幂等键 |
| 行为不确定 | Agent 自己决定调什么工具、什么顺序 | Harness 边界、验证闸门 |
| 环境不确定 | 世界不会停下来等 Agent | 渐进发现、重试策略 |

## 六、感知/记忆如何配合——最容易混淆的一对

| | 感知 | 记忆 |
|---|---|---|
| 角色 | 入口闸 | 存储层 |
| 关心 | 空间——这一轮看到什么 | 时间——窗口关后怎么沉淀 |
| 回答 | 什么信息有资格进入上下文 | 信息进来之后怎么保留 |

一份长期保留的 CLAUDE.md 既是感知（决定这轮看到什么），也是记忆（跨会话留下来）。

## 七、快速评估清单

每次迭代结束前对照检查：

- [ ] 五个地基（主循环/上下文/工具/状态/治理）是显式机制还是靠 prompt？
- [ ] 感知层有没有上下文分诊？信息按 P0/P1/P2/P3 分级了吗？
- [ ] 语义压缩有 Anchor 吗？excluded_approaches 记录了吗？
- [ ] 记忆有分层吗？草稿纸和长期记忆分开存了吗？
- [ ] 长任务有进度追踪？中断了能恢复吗？
- [ ] 失败被记录成可召回的教训了吗？还是只打了日志？
- [ ] Compound Error：步数能减少吗？单步质量能提高吗？
- [ ] 机械状态（id/金额/路径）是程序维护还是 LLM 拼接？
- [ ] 每个 Heavy 功能有对应的双轴坐标吗？
- [ ] 错误传播路径是什么？测试策略跟上了吗？
